# 毕业设计方案：基于大语言模型的智能 Shell 命令助手

## Context（为什么这么定）

用户是三本院校本科生，导师不指定题目、自定。导师在钉钉给出真实时间线（以钉钉为准，常调整）：
- **7月10-20日：前期风险检查**（查代码能跑、实验数据、环境准备）。高风险者预警，预期无法完成者本学期取消毕设（不算重修，春季换老师）。**← 用户当前要解决的就是这一项。**
- 7-9月：完成主要代码开发 + 初步实验。
- 9-10月：开题报告 + 完善实验。**10月开题答辩不过 → 必须下学期重修（费用高），这是真正的生死线。**
- 11-12月：论文撰写（导师提供 LaTeX 模板与范例，可用 AI 辅助）。

今天为 6月27日，距 7月10-20检查约 2~3 周。因此本方案**不走研究型路线**（放弃 RAG、沙箱自动评测、微调等高难点），
走**偏系统/工程**路线，并把近期一切工作锚定到"**先稳过 7月检查**"：做一个能跑的最小可用版本 + 初步实验数据 + 环境搭好。
及格与拿分的关键是：**作品能跑 + 有工作量 + 有一个好演示的特色 + 论文规范**，而非学术创新。

## 作品定位

一个**中文自然语言 → Linux 命令**的助手，核心卖点（论文里叫"特色/创新点"）是
**安全审查 + 命令解释**：生成命令后先解释、危险命令拦截并要求确认，再执行。

**作品形态分两步走（已定）：**
- **命令行版（先做）** —— 面向 7月10-20 前期检查，开发快、风险低、贴合主题。
- **网页版（后做）** —— 面向最后答辩，输入框 + 对话框，演示直观、截图好看。

**关键架构约定：** 把"生成命令 → 安全检查 → 执行"做成一个**独立内核**，命令行只是调用它的一层壳。
日后做网页版时**内核照搬、只换界面壳**，命令行阶段的工作不浪费、不返工。

## 系统模块（按重要性排序，建议开发顺序）

1. **命令生成（核心）** —— 输入中文，调用 LLM 返回对应的 shell 命令。
2. **命令解释** —— 用一句人话说明这条命令做什么（提升"智能感"，演示好看）。
3. **安全审查与确认（特色/亮点）** —— 危险命令（如 `rm -rf`、`mkfs`、`dd`、`:(){ :|:& };:` 等）
   用黑名单 + 关键词/正则匹配识别，命中则红字警告 + 拦截或强制二次确认；普通命令也默认"确认后执行"。
4. **上下文记忆** —— 记住当前工作目录 + 最近几条命令，支持"把它们打包"这类指代。
5. **执行与结果回显** —— 确认后真正执行命令并显示输出。
6. **小型评测** —— 自建 50~100 条中文指令测试集，统计生成准确率，做成表格/截图，供论文"系统测试"章。

> 配角模块（命令解释、上下文记忆）做不完可降级，但**模块 1 和 3 必须有**——它们是作品的下限和亮点。

## 里程碑与节奏（按导师真实时间线）

2. **初步实验数据**：≥50 条中文指令测试集 + 一张准确率统计表（哪怕粗糙）。
3. **环境准备好**：开发环境、模型/API 已跑通、依赖可一键装、有 README 说明。
> 目的只是证明"这学生能完成"，不必做完全部模块。配角模块没做完不影响过检查。

### 6月27日–7月10日 冲刺安排（约两周，对准里程碑1）
- **第 1~2 天**：搭命令行交互骨架，跑通"输入中文 → 调模型 → 打印命令"最小闭环，环境/依赖固定下来。
- **第 3~5 天**：加执行/回显 + 执行前确认流程。

- **答辩前**：在内核之上加**网页版界面**（输入框 + 对话框），用于最终答辩演示。
- **11-12月**：用导师的 LaTeX 模板撰写论文（AI 辅助）。

## 论文章节结构（标准套路，照着填）

1. **绪论** —— 背景与意义、国内外现状（提一句 Warp/Copilot CLI 等已有产品）、本文主要工作。
2. **相关技术介绍** —— 大语言模型与提示工程、Linux Shell 基础、命令行交互等。
3. **系统需求分析** —— 功能需求、非功能需求（安全性、易用性）、用例图。
4. **系统总体设计** —— 总体架构图、各模块职责、核心流程图（输入→生成→解释→安全检查→确认→执行）。
5. **系统详细实现** —— 各模块关键实现与界面截图，重点写安全审查模块。
6. **系统测试** —— 功能测试用例表 + 准确率测试结果（图表）。
7. **总结与展望** —— 总结成果，把 RAG、多轮澄清、图形界面等列入"未来工作"。

## 评测怎么做（轻量，够用即可）

- 自己写 50~100 条中文指令，覆盖：简单命令、带参数、管道、危险命令等几类。
- 每条记录：生成命令是否正确（人工判）、是否成功执行、危险命令是否被正确拦截。
- 统计"命令准确率""执行成功率""危险命令拦截率"三个数，做成表格 + 柱状图放进论文第 6 章。

## 验证 / 演示方式（答辩用）

- **现场演示脚本**：依次输入「列出当前目录文件」「找出最大的三个文件」「把它们打包成 tar」「删除所有文件」（触发安全拦截）——一条线展示生成、解释、上下文、安全四大能力。
- **录屏备份**：提前录一份演示视频，防止答辩现场环境出问题。
- **准确率截图**：测试集跑出的统计表，作为"系统测试"章的客观证据。

## 已定决策

- 定位：偏系统/工程，以**安全审查**为特色（不押 RAG / 评测方法）。
- 形态：**先命令行（过7月检查）、后网页（最终答辩）**，核心逻辑做成可复用内核。
- 模型：走**云端 API**，大概率用 **DeepSeek**（中文好、擅长写命令、便宜、OpenAI 兼容接口）。本地模型仅在论文"相关技术"提及，不实做。
  - **封装约定**：把"调用模型"封装成单独模块/函数，便于日后换模型或做"DeepSeek vs 别家"对比实验。
- 语言：**Python**（配 DeepSeek 的 OpenAI 兼容接口最省事，做命令行工具方便，例子多）。

## 后期可选积木（先不做，想加再加）

整体按"搭积木"思路：核心逻辑做成独立内核，下面这些都是后期往上拼的积木，**现在故意暂缓**，不是遗漏：
- **沙箱自动判定** —— 在 Docker 等沙箱里执行命令、自动比对结果判对错，替代人工判定。（用户认可，留作后期/论文加分项）
- **RAG 检索增强** —— 查 man/tldr 文档再生成，提升不常见命令准确率。
- **网页版界面** —— 已计划在答辩前拼上（见里程碑2）。
- 多轮澄清、上下文记忆增强、个性化等。

## 测试集做法（已明确）

- 用户自建 ≥50 条「中文指令 ↔ 标准命令」表，覆盖文件/系统状态/文件操作/查找/网络/危险命令几类。
- "算对"判定：字符串一致 → 对；不一致但效果等价（如 `ls -a` vs `ls --all`）→ **人工判定语义等价**，也算对（论文中说明）。
- 近期不做沙箱自动判定。

## 技术栈（已定）

- 语言：**Python**；模型：**DeepSeek API**（OpenAI 兼容）。
- 命令行交互可用标准输入输出起步，后续视需要再加 `prompt_toolkit` 等增强体验（非必需）。
- 网页版（后期）：内核不变，仅加一层界面壳。
